# External Auditor Request

We need an independent pre-mainnet review of `accord-protocol/accord-protocol`.

The review should focus on exact artifact identity, mainnet safety gates, compiled ErgoTree bytes, Base/EVM bytecode identity, and whether the docs describe the remaining verifier, wallet, bridge, and facilitator assumptions honestly.

## Handoff Commands

Maintainers prepare the review packet from a clean branch:

```bash
npm run audit:check
npm run audit:handoff -- --out dist/audit-handoff
```

The generated `dist/audit-handoff/audit-handoff.manifest.json` records:

- repository and commit;
- dirty working tree flag;
- every handoff file path;
- byte size for every file;
- sha256 hash for every file;
- summaries of `AUDITED_ERGOTREES.json` and `AUDITED_CONTRACTS.json`.

If the handoff is marked dirty, treat it as a draft only.

## Ergo Trees In Scope

Please verify that each `ergoTreeHex` in `packages/ergo-agent-scripts/data/AUDITED_ERGOTREES.json` corresponds to the reviewed source and enforces the intended semantics.

Trees:

- `task_hash_v0`
- `credential_v0`
- `chaincash_reserve_v0`
- `chaincash_receipt_v0`
- `chaincash_note_v0`
- `basis_reserve_v0`
- `basis_token_reserve_v0`

Primary Ergo questions:

1. Can any path drain Reserve collateral outside intended redemption, top-up, or mint rules?
2. Can expiry be bypassed or interpreted differently from the spec?
3. Can an attacker replace or mismatch the task hash because of off-chain/on-chain encoding differences?
4. Can a valid task output be copied from the mempool and used to steal redemption?
5. Do ChainCash Reserve, Note, and Receipt scripts compose safely in one redemption transaction?
6. Does Basis AVL state prevent double redemption across first and later redemptions?
7. Does the SDK enforce audited tree identity on mainnet, or merely non-empty trees?

## Base/EVM Scope

Please verify the `agentpay-base` safety model even if no Base mainnet deployment is yet approved.

Primary Base/EVM questions:

1. Does `AUDITED_CONTRACTS.json` bind network, address, and runtime bytecode hash tightly enough?
2. Does `agentpay-base` refuse mainnet writes without a positive audit policy unless `dangerouslyAllowUnauditedContract` is explicitly set?
3. Are runtime bytecode hash checks performed against deployed bytecode, not source-only claims?
4. Are empty or unknown manifests fail-closed?

## Transport And Policy Scope

Please review:

- Accord/402 replay behavior and agreement/payment id binding;
- Accord/MCP argument injection and stripping behavior;
- conformance L3 default-deny probes in `packages/accord-conformance/src/l3-security.ts`;
- buyer policy precision, concurrency, recipient allow-list, and approval-flow behavior.

## Required Deliverable

Please produce a signed report with:

- commit SHA reviewed;
- handoff manifest hash reviewed;
- list of artifacts reviewed;
- list of trees and contracts reviewed;
- source hash, post-template source hash, tree hash, or bytecode hash for each reviewed artifact;
- compiler/interpreter/tool versions used;
- all findings with severity;
- explicit accepted residual risks, if any;
- explicit mainnet recommendation: approved, conditionally approved, or rejected.

## Manifest Signing Output

If any artifact is approved for mainnet, the auditor should sign the relevant manifest as described in [`SIGNING_PLAYBOOK.md`](./SIGNING_PLAYBOOK.md).

The maintainer must not flip `mainnetAllowed: true` for an artifact unless:

- the auditor report covers the exact source and artifact hash;
- the auditor signs the manifest;
- the signed manifest verifies with the auditor's published public key;
- `docs/status.md`, `SECURITY.md`, and release notes are updated.

## Known Pre-Audit Concerns To Verify

- `task_hash_v0` is a pure hash predicate and appears front-runnable after `taskOutput` is revealed.
- `chaincash_reserve_v0` source comments previously contained TODOs for R5 preservation in top-up/mint paths.
- Basis reserve uses insert/update branching as an `insertOrUpdate` substitute because the current compiler does not expose `insertOrUpdate`.
- SDK high-level safety checks must require audited artifact identity on mainnet.
- Dangerous override flags must remain explicit, noisy, and absent from examples.
